# 技术方案

## 1. 目标

用 OpenClaw 做一个小型排障助手，重点不是“大而全”，而是把这条链路做通：

```text
输入报错 -> 读取真实仓库 -> 理解上下文 -> 执行真实复现 -> 输出修复建议
```

并且遵守以下边界：

- 不用 mock
- 不用假数据
- 不做静态占位结果
- 不把建议执行写成已执行

## 2. 完成标准

半天内完成，达到下面标准就算完成：

- 可以提交仓库地址、报错日志、触发步骤
- 可以拉取真实仓库
- 可以识别仓库技术栈
- 可以定位关键文件
- 可以输出报错解释
- 可以执行真实运行命令
- 可以生成一份结构化报告

## 3. 系统结构

### 3.1 入口层

入口保持简单，但数据流必须真实：

- `CLI`
  通过命令行传入仓库地址和报错文件

如果后续需要，再补一个极简 Web 页面。

### 3.2 仓库分析层

负责：

- 扫描文件树
- 识别语言与包管理器
- 定位入口文件
- 定位依赖文件和配置文件

判定规则建议：

- 有 `package.json` 则优先判为 Node.js
- 有 `requirements.txt` 或 `pyproject.toml` 则优先判为 Python
- 有 `Dockerfile` 时将其加入复现参考

### 3.3 报错分析层

负责：

- 对报错做粗分类
- 区分依赖问题、配置问题、启动问题、逻辑错误
- 提炼关键词

输出格式建议：

```json
{
  "error_type": "",
  "summary": "",
  "keywords": []
}
```

### 3.4 复现执行层

负责：

- 根据技术栈推断安装命令
- 根据常见入口推断启动命令
- 执行最小复现步骤
- 捕获 stdout、stderr 和退出码

示例：

- Python:
  - `pip install -r requirements.txt`
  - `python main.py`
- Node.js:
  - `npm install`
  - `npm run dev`

### 3.5 输出层

统一输出 Markdown 报告。

报告结构建议固定为：

1. 报错解释
2. 仓库摘要
3. 相关文件
4. 真实复现结果
5. 高概率根因
6. 修复建议
7. 验证步骤

## 4. 推荐实现文件

### `main.py`

负责主流程：

- 读取输入
- 调用各模块
- 汇总结果

### `repo_reader.py`

负责：

- 扫描仓库
- 输出技术栈和关键文件

### `analyzer.py`

负责：

- 解析报错
- 给出问题分类和关键词

### `reproducer.py`

负责：

- 生成实际执行命令
- 执行命令并记录结果

### `reporter.py`

负责：

- 把结果输出成 Markdown 报告

## 5. 半天内的取舍原则

为了半天内做完，但仍然保持真实数据流，取舍如下：

- 先不用数据库
- 先不用账号系统
- 先不用复杂前端
- 先不用多仓库并发
- 先不用真实自动补丁写回

把时间集中在：

- 真实仓库读取
- 仓库识别
- 报错解释
- 真实复现执行
- 报告输出

## 6. 4 小时时间分配

### 第 1 小时

- 搭项目目录
- 写输入结构
- 写仓库扫描

### 第 2 小时

- 写报错分类
- 写技术栈识别
- 写关键文件定位

### 第 3 小时

- 写复现命令生成与真实执行
- 写报告生成器

### 第 4 小时

- 跑两个案例
- 修正输出
- 准备截图和演示说明
