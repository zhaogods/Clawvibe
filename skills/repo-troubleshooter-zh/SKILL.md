# 仓库排障助手（中文）

收到与仓库报错、构建失败、安装失败、测试失败、启动失败或命令执行失败相关的请求时，使用此技能。

必须只基于真实仓库、真实错误日志、真实命令执行和真实输出进行分析。不要臆造复现结果、日志、根因、修复状态或验证结论。

## 触发信号

当用户请求中出现以下任一特征时，优先触发本技能：

- 分析本地仓库或 Git 仓库为什么跑不起来
- 根据一段错误日志、报错文本、stdout 或 stderr 定位根因
- 复现失败命令并输出排障结论
- 检查安装、构建、测试、启动、发布、脚本执行失败原因
- 明确要求“先分析，不要直接改仓库”
- 明确要求输出结构化排障报告

高频触发表达：

- “帮我看这个仓库为什么跑不起来”
- “这个项目安装依赖时报错”
- “这个 repo 构建失败，帮我定位原因”
- “根据这段错误日志分析根因”
- “复现这个失败命令并给出排障报告”
- “检查这个本地项目为什么测试不过”
- “不要直接改仓库，先分析问题”

## 必须遵守的行为

- 只使用用户提供的本地仓库路径或 Git URL。
- 只使用用户提供的原始错误日志、错误文本或错误文件。
- 只使用用户提供的触发步骤、命令或步骤文件。
- 必须在运行目录中的快照副本里执行，不直接修改源仓库。
- 必须捕获真实的 stdout、stderr、退出码和前置检查结果。
- 必须区分“已验证结论”“高置信推断”“信息不足”。
- 缺少关键输入时，先明确指出缺失项，不要编造补全。
- 不要声称已经修复，除非真实执行和验证步骤已经成功通过。
- 不要把建议性修复描述成已完成操作。
- 不要跳过真实执行后直接凭经验下结论，除非环境明确不支持执行。

## 不应误用的场景

以下情况不要把结论写成“已复现”或“已验证”：

- 用户没有提供仓库路径或 Git URL
- 用户没有提供原始错误日志、错误文本或错误文件
- 用户没有提供触发步骤、命令或步骤文件
- 当前环境不支持执行命令
- 入口脚本缺失、依赖不满足、仓库不可访问或输出目录不可写

如果发生上述情况，只能输出“缺少关键输入”或“未执行，仅基于现有信息进行初步分析”。

## 所需输入

确保任务尽可能包含以下输入：

- 仓库路径或 Git URL
- 原始错误日志文本或错误文件
- 原始触发步骤、命令字符串或步骤文件
- 可选分支名
- 可选输出目录

支持参数：

- `--repo`
  本地仓库路径或 Git URL
- `--error-log`
  原始错误文本
- `--error-file`
  包含原始错误文本的文件
- `--steps`
  原始触发命令或步骤描述
- `--steps-file`
  包含触发步骤的文件
- `--branch`
  可选远程分支
- `--runs-dir`
  可选输出目录

## 执行原则

### 1. 先做输入确认

在开始执行前，必须确认以下内容是否存在：

- 仓库输入是否存在且可访问
- 错误日志是否存在
- 触发步骤是否存在
- 入口脚本是否存在
- 输出目录是否可写

如果其中任一关键输入缺失，明确指出缺失项，并停止把结论写成“已复现”或“已验证”。

### 2. 必须运行真实流程

默认使用主入口脚本执行统一排障流程。

Python 入口：

```bash
python skills/repo-troubleshooter-zh/scripts/analyze_repo.py --repo <repo> --error-file <error.txt> --steps "<command>"
```

Windows PowerShell：

```powershell
.\skills\repo-troubleshooter-zh\scripts\run-skill.ps1 --repo <repo> --error-file <error.txt> --steps "<command>"
```

Windows cmd：

```cmd
skills\repo-troubleshooter-zh\scripts\run-skill.cmd --repo <repo> --error-file <error.txt> --steps "<command>"
```

Linux / macOS：

```bash
sh skills/repo-troubleshooter-zh/scripts/run-skill.sh --repo <repo> --error-file <error.txt> --steps "<command>"
```

如果用户提供的是原始错误文本而不是文件，可先写入临时错误文件再执行。

### 3. 只读取真实结果

执行后，优先读取并基于以下输出组织结论：

- `result.json`
- `final_report.md`

不要跳过执行直接根据经验下结论，除非环境明确不支持执行。在这种情况下，必须清楚标注为“未执行，仅基于用户提供信息进行初步分析”。

### 4. 输出必须分层

对外输出时，尽量按以下层次组织：

- 问题概述
- 前置检查
- 真实执行结果
- 错误分类
- 根因分析
- 修复建议
- 验证步骤
- 风险和未确认项

始终明确哪些内容来自真实执行，哪些内容属于基于日志和代码上下文的推断。

## 分析要求

### 前置检查

检查并记录：

- 仓库获取方式（本地路径或克隆）
- 分支信息（如有）
- 运行目录或快照目录
- 依赖管理工具（如 `npm`、`pnpm`、`yarn`、`pip`、`poetry`、`maven`、`gradle`）
- 运行时环境信息（如 Python / Node / Java 版本）
- 是否成功进入仓库
- 是否找到关键配置文件

### 错误分类

将问题尽量归类到一种或多种类型：

- 依赖缺失或版本冲突
- 解释器或运行时版本不匹配
- 环境变量缺失或配置错误
- 构建脚本失败
- 测试失败
- 路径错误或文件缺失
- 权限问题
- 网络或下载失败
- 编译失败
- 框架配置错误
- 包管理器锁文件或缓存问题
- 平台兼容性问题
- 用户命令本身错误

### 根因分析

根因分析必须优先引用真实证据，例如：

- 关键报错行
- 退出码
- 缺失文件名
- 缺失模块名
- 版本冲突信息
- 失败命令及其上下文

不要只给泛泛建议，例如“检查环境配置”。要尽量落到具体对象、具体命令、具体文件、具体版本。

### 修复建议

修复建议应尽可能具备以下特征：

- 具体
- 可执行
- 与当前错误直接对应
- 能区分“首选修复”和“备选修复”
- 说明可能影响

建议形式优先覆盖：

- 修改哪一个配置项
- 安装哪一个依赖
- 切换到哪个运行时版本
- 重新生成哪一个锁文件
- 用哪条命令验证修复是否生效

### 验证步骤

必须提供可复验的验证步骤，优先使用最小闭环，例如：

- 重新安装依赖
- 重新执行失败命令
- 检查退出码
- 检查目标文件、服务或测试结果是否恢复正常

不要把“建议用户验证”写得过于笼统。

## 输出要求

主要输出文件：

- `result.json`
- `final_report.md`

对用户回答时，优先总结以下内容：

- 是否成功复现
- 最可能根因
- 证据是什么
- 建议先做哪一步修复
- 修复后如何验证

如果输出中存在不确定项，明确写出：

- 哪些项已经被真实执行验证
- 哪些项仍是推断
- 哪些项因环境限制未验证

## 推荐回答风格

- 使用中文输出
- 先给结论，再给证据
- 尽量具体，不空泛
- 对未验证部分保持克制
- 不夸大修复把握
- 不把“可能”说成“已经确认”

## 限制说明

- 该技能优先支持 Python 和 Node 仓库。
- 对其他语言或构建系统，如果脚本能正常运行，也可继续使用，但必须如实报告支持程度。

- 如果仓库不可访问、脚本缺失、依赖不满足或运行失败，应明确报告阻塞点。
