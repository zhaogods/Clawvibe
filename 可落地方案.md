# 可落地方案

## 项目主题

**让新手也看懂报错：我用 OpenClaw 做了个会读仓库、会复现、会给补丁的 vibe coding 助手**

## 1. 项目目标

做一个小而完整的 OpenClaw 应用，解决一个非常具体的问题：

> 当新手面对陌生仓库和报错时，系统能够自动读取仓库、组织排查、尝试复现，并给出修复建议或补丁草案。

这个项目的重点不是“多聪明”，而是：

- 能跑
- 能解释
- 能定位
- 能给出下一步操作

并且从第一版开始就遵守以下原则：

- 接真实仓库
- 读真实日志
- 执行真实命令
- 输出真实运行结果
- 不使用假数据和占位结果

---

## 2. 解决的问题

新手处理开源项目报错时，通常会卡在 4 个地方：

1. 看不懂报错内容
2. 不知道仓库里哪个模块相关
3. 不知道先执行什么命令来复现
4. 就算知道大概原因，也不会给出修改方案

这个项目直接围绕这 4 个问题设计，不扩散到别的方向。

---

## 3. 使用流程

用户只需要提供 3 类信息：

- 仓库地址
- 报错日志
- 触发步骤

系统输出 6 类结果：

- 报错解释
- 仓库结构摘要
- 相关文件定位
- 真实复现记录
- 根因判断
- 修复建议或补丁草案

完整链路如下：

```text
输入仓库地址和报错
  -> OpenClaw 拉取或读取仓库
  -> 扫描项目结构与依赖
  -> 分析报错与相关代码
  -> 组织复现命令
  -> 捕获失败日志
  -> 输出原因解释、修复建议、验证步骤
```

## 4. 功能设计

### 4.1 报错解释

作用：

- 把原始报错翻译成新手能看懂的话
- 解释错误属于哪一类
- 说明它通常意味着什么

输出示例：

- 这不是代码逻辑错误，而是依赖未安装
- 这不是依赖缺失，而是配置文件路径不对
- 这一步失败发生在服务启动阶段，不是在业务执行阶段

### 4.2 仓库阅读

作用：

- 识别技术栈
- 找出入口文件
- 找出依赖文件和配置文件
- 提取可能相关的模块

优先识别的文件包括：

- `README.md`
- `package.json`
- `requirements.txt`
- `pyproject.toml`
- `Dockerfile`
- `.env.example`
- 入口脚本

### 4.3 复现组织

作用：

- 根据仓库类型自动生成安装和运行命令
- 在真实环境中按最小步骤执行
- 捕获真实失败日志

示例：

- Python 项目优先执行 `pip install -r requirements.txt` 与实际入口脚本
- Node 项目优先执行 `npm install` 与项目声明的启动命令

### 4.4 根因分析

作用：

- 把报错、目录结构、配置、运行结果关联起来
- 判断是环境问题、依赖问题、配置问题还是代码问题
- 给出高概率原因排序

### 4.5 修复建议

作用：

- 给出建议修改点
- 指出修改文件
- 说明为什么这样改
- 给出基于真实文件位置的补丁草案

### 4.6 验证步骤

作用：

- 告诉用户改完后如何确认问题修好
- 输出验证命令
- 指出预期结果

## 5. 系统结构

建议按 5 个模块实现。

### 模块 1：任务接收

输入：

- 仓库地址
- 报错日志
- 触发步骤

输出：

- 标准任务对象

### 模块 2：仓库分析

输入：

- 本地仓库目录

输出：

- 项目类型
- 入口文件
- 依赖文件
- 配置文件
- 候选相关文件

### 模块 3：复现执行

输入：

- 项目类型
- 实际执行命令

输出：

- 运行日志
- 错误日志
- 是否成功复现

### 模块 4：问题诊断

输入：

- 报错内容
- 仓库结构
- 复现结果

输出：

- 问题类别
- 高概率根因
- 建议检查文件

### 模块 5：修复输出

输入：

- 诊断结果

输出：

- 修复建议
- 补丁草案
- 验证步骤
- 汇总报告

## 6. OpenClaw 在项目中的角色

OpenClaw 不是只负责聊天，而是负责实际执行整条链路：

- 接收任务输入
- 读取本地或远程仓库
- 调用终端命令
- 分阶段处理任务
- 输出结构化结果

这个项目适合 OpenClaw 的原因很直接：

- 它需要读文件
- 它需要跑命令
- 它需要理解代码上下文
- 它需要生成可执行建议

## 7. 技术实现建议

### 7.1 前端入口

入口可以保持简单，但数据流必须真实：

- 简单 Web 表单
- OpenClaw 聊天式输入页

表单字段只保留：

- 仓库地址
- 报错日志
- 触发步骤

### 7.2 后端任务流

按顺序执行：

1. 拉取仓库
2. 扫描结构
3. 识别技术栈
4. 推断运行命令
5. 执行真实复现
6. 分析错误
7. 生成修复建议
8. 生成报告

### 7.3 结果格式

统一输出为 Markdown 或 JSON，再渲染给用户。

推荐结构：

```json
{
  "error_explanation": "",
  "project_type": "",
  "related_files": [],
  "reproduce_steps": [],
  "root_causes": [],
  "fix_suggestions": [],
  "verify_steps": []
}
```

## 8. 部署方案

比赛环境直接面向阿里云部署。

### 8.1 部署位置

- 阿里云 ECS

### 8.2 部署内容

- OpenClaw
- 项目服务脚本
- 仓库分析模块
- 日志目录
- 简单页面或命令行入口

### 8.3 运行方式

用户提交一个任务后：

- 系统在工作目录创建一个任务文件夹
- 拉取目标仓库
- 执行分析和复现
- 输出报告到该任务目录

任务目录建议如下：

```text
runs/
└─ task_001/
   ├─ input.json
   ├─ repo_summary.json
   ├─ reproduce.log
   ├─ diagnosis.json
   ├─ patch.diff
   └─ final_report.md
```

## 9. 开发顺序

直接按这个顺序做，不要同时摊太大。

### 第一步

先做输入、仓库扫描、报错解释。

做完后要能实现：

- 输入仓库地址和报错
- 拉取真实仓库
- 输出项目类型
- 输出关键文件
- 输出报错的人话解释

### 第二步

加上复现执行。

做完后要能实现：

- 自动推断基础运行命令
- 真正执行命令
- 捕获日志
- 输出复现是否成功

### 第三步

加上根因分析和修复建议。

做完后要能实现：

- 给出问题类别
- 指出建议修改文件
- 输出补丁草案

### 第四步

加上页面、留档和真实案例。

做完后要能实现：

- 可提交任务
- 可查看报告
- 可展示至少 2 个真实案例
