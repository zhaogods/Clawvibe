# 上下文导出

## 当前项目主题

**让新手也看懂报错：我用 OpenClaw 做了个会读仓库、会复现、会给补丁的 vibe coding 助手**

## 当前结论

- 这个目录的核心目标不是先写比赛论坛稿，而是先做一个可参赛、可运行的小项目。
- 比赛是阿里云体系下的 OpenClaw 挑战赛，项目本体比展示文章更重要。
- 当前最适合的项目方向已经固定为：
  - 面向新手开发者的开源仓库排障助手
- 不再继续扩写“内容工厂”方向作为主项目。

## 项目要解决的问题

新手在处理陌生开源仓库时，通常卡在下面几个问题：

1. 看不懂报错
2. 看不懂仓库结构
3. 不知道先执行什么命令
4. 不知道该改哪个文件
5. 不知道怎么验证修复是否成功

这个项目要做的是把这条链路缩短：

```text
报错出现 -> 理解报错 -> 读取仓库 -> 找相关文件 -> 给复现步骤 -> 给修复建议
```

## 已确定的项目范围

项目是一个小项目，不做大平台。

当前范围：

- 输入：
  - 仓库地址
  - 报错日志
  - 触发步骤
- 输出：
  - 报错解释
  - 仓库摘要
  - 相关文件定位
  - 复现命令建议
  - 根因判断
  - 修复建议
  - 补丁草案
  - 验证步骤

当前明确不做：

- 不做复杂前端
- 不做数据库
- 不做登录系统
- 不做自动提交 PR
- 不做全语言、全仓库类型支持
- 不做 mock
- 不做假数据
- 不做静态占位结果

## 当前建议的实现方式

优先做一个 `CLI` 版本，理由：

- 最省时间
- 半天内最容易完成
- 最适合先把主链路跑通

推荐模块：

- `app/main.py`
- `app/repo_reader.py`
- `app/analyzer.py`
- `app/reproducer.py`
- `app/reporter.py`

推荐目录：

```text
app/
├─ main.py
├─ analyzer.py
├─ repo_reader.py
├─ reproducer.py
├─ reporter.py

examples/
runs/
docs/
```

## 当前技术方案摘要

### 1. 仓库扫描

识别：

- Python / Node.js
- 依赖文件
- 配置文件
- 入口文件

优先检查：

- `README.md`
- `package.json`
- `requirements.txt`
- `pyproject.toml`
- `Dockerfile`
- `.env.example`

### 2. 报错分析

粗分类为：

- 依赖问题
- 配置问题
- 启动问题
- 逻辑错误

### 3. 复现执行

根据仓库类型生成实际执行命令，并捕获真实结果：

- Python：
  - `pip install -r requirements.txt`
  - `python main.py`
- Node.js：
  - `npm install`
  - `npm run dev`

### 4. 报告输出

统一输出 Markdown 报告，结构固定为：

1. 报错解释
2. 仓库摘要
3. 相关文件
4. 真实复现结果
5. 高概率根因
6. 修复建议
7. 验证步骤

## 当前时间约束

用户要求：

- 希望半天内开发完成

因此当前策略已经收缩为：

- 只做最核心闭环
- 先做 CLI
- 先做两个演示案例
- 先让报告能产出

建议的 4 小时顺序：

### 第 1 小时

- 搭项目目录
- 写输入结构
- 写仓库扫描

### 第 2 小时

- 写报错分类
- 写技术栈识别
- 写关键文件定位

### 第 3 小时

- 写复现命令生成
- 写报告生成器

### 第 4 小时

- 跑两个案例
- 修正输出
- 准备截图和演示说明
